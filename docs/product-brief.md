# ExpatTH 产品理解

本文是对当前 BP 的项目级理解，用作页面设计、API 反推、数据库重建和后续实现的产品依据。

原始 BP 位于：

```text
epmerchant/docs/businessplan.md
```

## 一句话定位

ExpatTH 是面向泰国市场的本地生活服务交易平台，核心不是信息黄页，而是帮助用户用更低筛选成本找到可信服务，并让服务商获得稳定、可管理、可复盘的订单来源。

## 要解决的问题

泰国本地服务市场的痛点集中在交易前和履约中：

- 服务信息分散在社交媒体、熟人推荐和线下渠道。
- 服务商质量、资质、响应速度和价格规则不透明。
- 用户很难判断服务是否可靠，也很难比较不同服务方案。
- 询价、沟通、下单、履约、评价没有统一链路。
- 平台缺少能沉淀信用、订单记录和服务标准的数据结构。

因此产品不能只做“展示商家”。真正要做的是把需求、服务标准、商家能力、订单履约和信用评价结构化。

## 产品主线

当前阶段应坚持以下主线：

1. 用户先选择平台标准服务或服务类别。
2. 平台根据服务类型决定进入固定报价、询价或咨询路径。
3. 用户提交结构化需求，而不是直接找某个商家私聊。
4. 平台或系统把需求转化为订单、询盘或服务请求。
5. 商家在商家端承接、报价、沟通、履约。
6. 平台后台负责审核、监控、纠纷处理、数据沉淀。

这个方向和当前规则一致：不要让用户旅程一开始就变成“选商家”。

## 三端边界

### 用户端 `ep`

用户端负责把复杂服务变成容易理解的购买或咨询路径：

- 服务分类与标准服务入口。
- 服务详情、要求模板、价格预估。
- 下单、询价、预约、订单记录。
- 用户资料、地址、钱包、评价和消息。
- 清晰展示平台保障、履约状态和下一步动作。

### 商家端 `epmerchant`

商家端负责让服务商管理供给和履约：

- 服务资料、服务范围、价格和可服务时间。
- 询盘/订单处理。
- 商家资料、认证和审核状态。
- 数据统计、广告推广和运营工具。
- 与平台标准服务的映射关系。

### 后端 `epbkend/expatth-backend`

后端负责可信交易和数据模型：

- 认证、权限、商家审核。
- 服务分类、标准服务、需求模板。
- 订单、询价、报价、履约状态。
- 评价、消息、钱包、支付意图。
- 地理位置检索、关键词搜索、AI 向量检索和智能推荐。

### 共享契约 `ep-shared`

`ep-shared` 是跨仓一致性的源头：

- 术语和业务边界。
- 状态机。
- API 请求池和接口注册表。
- 数据库规划。
- 前后端字段命名与错误码。

凡是涉及共享概念，先改 `ep-shared`，再落到后端和前端。

## 数据库方向

当前数据库应从 clean rewrite 的产品模型重新设计，不应直接导入旧 MySQL schema。

目标数据库是 PostgreSQL，并启用：

- PostGIS: 位置、服务半径、附近商家、区域覆盖。
- pgvector: 服务、需求、商家能力、用户意图的 embedding 检索。
- pg_trgm: 搜索关键词、服务名称、商家名称和多语言文本的模糊匹配。

优先建模对象：

- `users`
- `merchants`
- `merchant_profiles`
- `service_categories`
- `standard_services`
- `merchant_services`
- `requirement_templates`
- `service_requests`
- `quotes`
- `orders`
- `order_events`
- `reviews`
- `messages`
- `wallet_transactions`
- `search_documents` 或业务表 embedding 字段

旧表、旧字段、旧迁移只有在能服务新模型时才保留。不能解释清楚用途的表不进入新数据库。

## 页面优先级

先从用户端倒推设计：

1. 首页和标准服务入口。
2. 服务分类页。
3. 标准服务详情页。
4. 需求填写/询价/预约页。
5. 订单创建结果页。
6. 订单中心和订单详情。

商家端随后围绕“接单和管理服务供给”展开：

1. 商家登录/注册。
2. 商家认证状态。
3. 服务管理。
4. 询盘和订单处理。
5. 商家数据和推广。

后台最后处理审核、监管和运营配置。

## 当前阶段的判断

当前项目处于独立开发和设计阶段，允许推翻旧实现。最重要的是让页面逻辑、API 契约、数据库模型保持一致，不要让旧代码和旧表结构拖住产品形态。

短期目标：

- 以前端页面体验确认业务路径。
- 用 `ep-shared` 记录页面需要的 API 和数据对象。
- 用 PostgreSQL 重建干净的业务表。
- 再让后端提供稳定接口。

长期目标：

- 形成服务商信用体系。
- 形成标准化履约能力。
- 用搜索、地理位置和 AI 推荐提高服务匹配效率。
- 从 Web 验证后再考虑移动端。
